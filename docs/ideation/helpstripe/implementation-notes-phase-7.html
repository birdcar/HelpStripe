install:broadcasting — JS deps installed with bun, .env.example placeholders

Context
The spec calls for php artisan install:broadcasting --reverb --no-interaction. The command published config/broadcasting.php + config/reverb.php, wired channels: …/routes/channels.php into bootstrap/app.php, scaffolded resources/js/echo.js + import './echo' in the (previously empty) resources/js/app.js, and added REVERB_* / VITE_REVERB_* to .env — but its final "Installing and building Node dependencies" step crashed with TTY mode requires /dev/tty (it shells out to npm install, and there is no TTY in this session).
Decision
The PHP-side scaffold all landed correctly, so the only gap was the JS packages. Installed them with the repo's package manager: bun add laravel-echo pusher-js (the project uses bun, not npm). Then added the REVERB_* / VITE_REVERB_* block to .env.example with empty APP_ID/KEY/SECRET placeholders (the real generated values live only in the untracked .env) and flipped BROADCAST_CONNECTION to reverb there. bun run build then bundled Echo + Pusher into the manifest (app.js grew from 0 B to ~73 KB).
Alternative
Re-run the installer's node step under a pseudo-TTY, or commit the generated secrets to .env.example. Rejected: bun is the project standard (npm would desync the lockfile), and example env files must never carry real credentials.

ChannelAuthTest — driving /broadcasting/auth under the null broadcaster

Context
The spec's playground is "ChannelAuthTest posting to /broadcasting/auth." The suite default (phpunit.xml) is BROADCAST_CONNECTION=null. The null broadcaster does two unhelpful things for this test: it returns a non-JSON 200 for any auth request (so even denials look like successes), and — the subtler problem — the framework never loads the channel authorization closures from routes/channels.php when the active driver is null, leaving the channel registry empty. With an empty registry, every channel name fails to match and Laravel throws a bare AccessDeniedHttpException — so a legitimate team member 403s for the wrong reason.
Decision
In a beforeEach, point broadcasting at the reverb connection (the Pusher-protocol driver that actually runs the closure and signs the response) with dummy credentials, then require base_path('routes/channels.php') so the closures register against that live driver. The member case then returns 200 with a channel_data JSON string carrying user_id (a string, per the Pusher protocol) and user_info = the closure's array. Foreign-team and guest cases 403 through the real authorization path. Nothing connects to a server — the driver only signs.
Alternative
Invoke the channel closure in isolation (resolve it out of the broadcaster and call it). Rejected: it skips the /broadcasting/auth route, the guard that denies guests, the route-model binding of {helpdeskRequest}, and the presence-payload shaping — i.e. most of what could break. Driving the real endpoint is the higher-fidelity test the spec asked for.

Viewer banner is a Blade partial, not a Livewire SFC

Context
The spec lists the new file as resources/views/pages/requests/⚡viewers.blade.php — the ⚡ prefix denotes a Livewire single-file component in this repo. But a presence channel's membership is per-subscription: if a standalone ⚡viewers component also joined request.{id}, each browser would open a second subscription and register the same user twice, double-counting the roster (ghost viewer).
Decision
Kept presence in exactly one place — the ⚡show component owns the channel listeners (getListeners()) and the $viewers state — and made the banner a pure presentational Blade partial at resources/views/pages/requests/viewers.blade.php (no ⚡), @included with the $viewers array. It holds no state and joins no channel, so the ⚡ prefix would have been misleading. All four presence/refresh handlers and dedupe-by-id / self-exclusion live in ⚡show.
Alternative
A nested ⚡viewers Livewire component receiving $viewers as a prop (no channel of its own). Rejected: it adds a component round-trip for zero benefit — the banner only ever renders parent state and has no interactions. A partial is the honest, lighter representation; the filename drops ⚡ to signal "not a component."

getListeners() over the #[On('echo-presence:…')] attribute

Context
The spec's Open Item asked to verify the Livewire 4 echo-presence listener syntax. The static #[On('echo-presence:request.{id},here')] attribute interpolates class property names (e.g. {order.id}) — but our channel id comes from the bound $helpdeskRequest model, and we need four listeners (here / joining / leaving / NoteAdded) keyed off the same id.
Decision
Used getListeners() (Boost-confirmed for Livewire 4), building "echo-presence:request.{$this->helpdeskRequest->id},…" at runtime for all four signals. NoteAdded is a class-name event, so no leading dot is needed (the dot prefix is only for broadcastAs() custom names).
Alternative
The dynamic attribute form #[On('echo-presence:request.{helpdeskRequest.id},here')]. Workable for one listener, but getListeners() keeps all four declarations in one obvious place and matches the existing repo pattern (⚡save-filter-modal uses attributes for static events; dynamic channel ids belong in getListeners).
