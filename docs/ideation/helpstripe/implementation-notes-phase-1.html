Implementation Notes — Phase 1

HelpStripe · 2026-06-10 · Foundation & Domain Models

User model — trait collision between HasTeams and spatie HasRoles

Context
The spec said "Add HasRoles to User". spatie/laravel-permission v8 (the version composer resolved) added its own teams() relation to HasRoles, which collides with the starter kit's HasTeams::teams(). PHP refuses to compose the traits without explicit resolution.
Decision
Trait conflict resolution: HasTeams::teams insteadof HasRoles keeps the starter's team relation as teams(); spatie's is aliased to permissionTeams(). Spatie's teams feature is disabled ('teams' => false), spatie never calls its own teams() internally, and the disabled-mode relation returns no rows — so the alias is inert. Bonus: this is a second real-world name-collision lesson for the tour doc.
Alternative
Renaming the starter's teams() relation — rejected: it's consumed across the starter kit (team switcher, settings screens) and would expand the diff far beyond the phase scope.

Request history — activitylog v5 stores diffs in attribute_changes

Context
The spec (and spatie v4 docs) assume attribute diffs live in activity.properties (['attributes' => ..., 'old' => ...]). Composer resolved spatie/laravel-activitylog v5.0, which moved model diffs to a dedicated attribute_changes JSON column; properties is now reserved for custom withProperties() data.
Decision
Tests and the tour doc read $activity->attribute_changes['attributes'] / ['old']. The trait also moved namespaces (Spatie\Activitylog\Models\Concerns\LogsActivity, Spatie\Activitylog\Support\LogOptions) — verified against vendor source.
Alternative
Pinning activitylog to v4 — rejected: v5 is the version that supports Laravel 13, and teaching the current API is the point of the repo.

Migrations — identical timestamps from batch make:model

Context
Running five php artisan make:model -mf commands in one shell invocation produced five migrations with the same timestamp. Laravel falls back to alphabetical order on ties, which would run create_notes_table before create_requests_table and fail on the foreign key.
Decision
Renamed the files with sequential timestamps (…531 customers, …532 categories, …533 mailboxes, …534 requests, …535 notes) so filename order equals dependency order. Called out in the tour doc as part of the migration-ordering lesson.
Alternative
Combining all tables into one migration file — rejected: per-table migrations match Laravel convention and teach ordering explicitly.

UserFactory — automatic personal-team creation disabled (user direction)

Context
The starter kit's UserFactory::configure() created a personal team per user via an afterCreating hook, which gave the demo seeder 5 teams instead of the single "HelpStripe Support" installation. The user directed: disable personal-team creation — this project doesn't need it.
Decision
Removed the automatic hook; the behavior is now an opt-in withPersonalTeam() factory state. Six starter test files that exercise personal-team behavior (auth redirects, team deletion fallbacks, dashboard) opt in explicitly, preserving their original preconditions. DemoSeeder went back to plain User::factory() — no more forceCreate workaround.
Alternative
Also removing personal-team creation from the Fortify registration flow (CreateNewUser still creates one) — deferred: the app's routes are team-scoped (/{current_team}/dashboard), so a team-less registrant would 403 after registering. What team a fresh registrant should get is a product decision for a later phase; CreateNewUser is the one remaining personal-team creation site.

Test runner — laravel/pao suppresses Pest output under agents

Context
The starter kit ships laravel/pao, which detects agent sessions and rewrites Pest output as a single JSON line — and emitted nothing for fatal bootstrap errors (the trait collision was invisible: exit 1, zero bytes of output).
Decision
Used PAO_DISABLE=1 when debugging to get real error output. Validation commands in CI paths (composer test, ./init.sh) left untouched.
Alternative
None needed — environment quirk, not a code decision.
