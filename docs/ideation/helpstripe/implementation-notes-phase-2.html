Implementation Notes — Phase 2

HelpStripe · 2026-06-10 · Ticket Management

Notifications — missing notifications table migration

Context
The spec calls for RequestAssignedNotification on the database + mail channels, but its File Changes list has no migration for Laravel's notifications table, which the database channel requires. No such migration existed from earlier phases.
Decision
Added php artisan make:notifications-table (2026_06_10_145654_create_notifications_table.php) — the stock framework migration, unmodified.
Alternative
Mail-only channel — rejected: the spec and tour doc explicitly demo the database-channel notification.

History tab — activitylog v5 renamed the subject relation

Context
The spec says the History tab renders $request->activities. In activitylog v5 (the installed major), the LogsActivity trait's relation is activitiesAsSubject() — activities() no longer exists.
Decision
Used activitiesAsSubject() in the ⚡show component. Continues Phase 1's v5 discovery (diffs live in attribute_changes, not properties).
Alternative
Adding an activities() alias relation on the model — rejected: shadowing a vendor rename invites confusion later.

Response picker — Livewire updated-hook instead of Alpine insertion

Context
The spec suggests the canned-Response picker "inserts body text via Alpine". Doing it client-side means shipping every response body into the page and entangling Alpine state with wire:model on the textarea.
Decision
Server-side updatedSelectedResponse() hook: picking a Response appends its body to the draft (team-scoped lookup, append not replace). Testable in Pest without a browser, and one less moving part.
Alternative
Alpine x-on:change writing to $wire.replyBody — works, but untestable in the Livewire test harness and duplicates state client-side.

ChangeStatus — resolved_at semantics for Closed and re-transitions

Context
The spec defines "Resolved sets resolved_at, back to Active clears it" but is silent on Closed and on Resolved → Closed.
Decision
Entering either non-open status (Resolved, Closed) stamps resolved_at once; Resolved → Closed keeps the original timestamp; entering any open status (Active, Pending) clears it. Matches the Phase 1 seeder, which gives closed requests a resolved_at, and keeps Phase 8's resolution metrics monotonic.
Alternative
Stamping only on Resolved — rejected: a request closed without passing through Resolved would never count as resolved in reports despite being finished.

Properties panel — category/urgency are direct updates, not actions

Context
The spec says "Property changes call the actions directly" but defines exactly four actions (create, note, assign, status) — none for category, urgency, or tags.
Decision
Category, urgency, and tags are plain model updates in the component; LogsActivity records category/urgency changes for the History tab either way. No later phase hooks these changes, so no event/action indirection is warranted.
Alternative
Inventing ChangeCategory/ToggleUrgent actions — rejected as scope expansion: nothing consumes them, and the spec's action list is explicit.

Save-filter modal — criteria delivered by event, not mount prop

Context
The modal is a nested SFC; mount props are bound once, but the queue's criteria change constantly after mount. The spec's referenced pattern (⚡create-team-modal, which lives at resources/views/components/, not under pages/teams as the spec implied) carries no parent state at all.
Decision
The queue dispatches save-filter-modal:open with the current criteria array; the modal's #[On] handler stores them and opens itself via Flux::modal()->show(). Also note: the starter's dispatch('close-modal', name: …) convention is a no-op with Flux v2 (its real events are modal-show/modal-close), so this phase uses the documented Flux::modal() PHP API instead.
Alternative
Re-keying the nested component on criteria changes to force re-mount — rejected: discards modal state mid-interaction and re-renders on every keystroke in the search box.
