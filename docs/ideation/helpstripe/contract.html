Context · 01

The problem

01A new support engineer joining HelpSpot must learn how a Laravel application works — not abstractly, but in the exact shape of the product they will support every day. Generic Laravel tutorials teach todo lists; this engineer needs requests, mailboxes, triggers, portals, and the vocabulary HelpSpot customers use on real tickets.

02HelpStripe closes that gap: a teaching repository that reimplements the six pillars from HelpSpot's public tour — Shared Inbox, Ticket Management, Automation Rules, Reporting, Knowledge Base, Self-Service Portal — in simple, idiomatic Laravel 13 on this repo's existing Livewire 4 + Flux + Fortify starter. Every pillar doubles as a Laravel lesson (Eloquent relations, the mail pipeline, events and the scheduler, aggregates, broadcasting, public routes), delivered through annotated source code and a per-pillar tour doc.

03Features must genuinely work and be demoable end-to-end on Laravel Herd — real email both directions via Resend, real websockets via Reverb — but are explicitly not production-ready. The bar is: follow the demo script and it works.

Commit · 02

Goals

1 Implement all six HelpSpot tour pillars in working, demoable form using HelpSpot's own vocabulary (Request, Filters, Knowledge Books, Responses, Mail Rules / Triggers / Automation Rules).
2 Real email round-trip via Resend: an inbound webhook creates Requests, agent replies thread correctly into the customer's inbox, and mail:replay reproduces the inbound flow offline.
3 Each pillar teaches distinct Laravel concepts through annotated source plus a docs/tour/ walkthrough a Laravel newcomer can follow file-by-file.
4 Use Spatie packages as the idiomatic backbone: laravel-permission (permission groups), laravel-activitylog (request history), laravel-webhook-client (Resend inbound), laravel-medialibrary (attachments), laravel-tags, laravel-sluggable.
5 Every phase keeps the repo harness green: ./init.sh (Pint + Pest) passes and feature_list.json tracks phase status with recorded evidence.

MVP ×5
MVP — must ship ×5
  • Domain foundation: Request, Customer, Category, Mailbox models, status enum, spatie/laravel-permission groups (Administrator, Help Desk Staff), factories and rich seeders— Every pillar builds on these models and the seeded demo data; one seeded team represents the HelpSpot installation.
  • Ticket Management: request queue + detail Livewire UI, single assignment, public reply vs private note, full history via activitylog, saved Filters— The heart of the product the engineer will support — and the tour's 'single assignment, full history' claims.
  • Shared Inbox / Resend email pipeline: outbound Mailables with threading headers, inbound webhooks via laravel-webhook-client, attachments via medialibrary, mail:replay offline fallback— HelpSpot is fundamentally an email product; this is the centerpiece integration and the richest Laravel lesson.
  • Self-Service Portal: public submit form, confirmation email, status lookup via email + access key (no customer accounts)— Completes the customer-side loop exactly the way real HelpSpot does it; teaches public vs authenticated route groups.
  • Teaching layer: annotated source code plus one docs/tour/ doc per pillar with a step-by-step demo script— The entire point of the repository — code without the tour fails the stated goal.
Full — target outcome ×5
  • Automation: Mail Rules (inbound pipeline), Triggers (request events), scheduled Automation Rules, with a DB-backed condition/action rule builder UI— All three HelpSpot automation layers; teaches events, listeners, queued jobs, and the scheduler.
  • Knowledge Base: Books → Chapters → Pages with drafts, slugs, portal browsing, and LIKE-based search— Faithful to HelpSpot's hierarchy; teaches nested relationships and route model binding.
  • Reporting: dashboard with three core reports rendered as Flux charts, plus per-category SLA first-response targets with breach flagging— Honors the tour's 'response times, agent performance, SLA tracking'; teaches Eloquent aggregates and date grouping.
  • Collision detection via Laravel Reverb presence channels and Echo on the request detail page— A distinctive HelpSpot behavior; presence channels are the exact primitive for it. Runs on Herd locally.
  • API intake channel: POST /api/v1/requests with token auth and an API resource response— Completes the tour's three intake channels and is the repo's only API-building lesson.
Out of scope — said no on purpose
  • Real IMAP/SMTP mailbox polling — Resend inbound webhooks replace it with less infrastructure and a better webhook lesson.
  • Multi-brand and multi-team features — One seeded team represents the single HelpSpot installation; team machinery stays but is never built against.
  • Custom fields, forums, spam filtering, time tracking, request merging, satisfaction surveys — Confirmed cut list — parked in Future Considerations; the harness makes adding them later cheap.
  • Business-hours SLA calendars — SLA stays simple wall-clock first-response targets; calendars add complexity without a new Laravel lesson.
  • Third-party integrations and Scout full-text search — LIKE search is enough for the demo; Scout is a future lesson.
  • Production hardening — Explicit non-goal: no deliverability tuning, scaling, or security review beyond framework defaults.
Future — someday, maybe
  • Knowledge Base deflection: suggest articles while a customer types a portal request
  • Request merging (transactions + relation reparenting lesson)
  • Time tracking with reporting rollups
  • Scout-backed full-text search
  • Custom fields
  • Satisfaction surveys
