Implementation Notes — Phase 6

HelpStripe · 2026-06-10 · Automation Rules

Mail rules accumulate create-payload overrides, not email mutations

Context
The spec described applyMailRules() as the seam and implied the engine would "recategorize, reroute, or drop the email." The Phase 3 seam signature was applyMailRules(InboundEmail): InboundEmail — a pass-through that returned a (possibly mutated) email. But mail-rule actions (set_category, assign_to, set_urgent) are properties of the request, not the email — there's no request yet when a mail rule runs.
Decision
Introduced MailRuleEvaluator::overridesFor(InboundEmail, teamId): array, which returns a {category_id?, assigned_to?, is_urgent?} overrides map. The job's signature changed to applyMailRules(InboundEmail, int $teamId): array and the new-request branch spreads the overrides into the CreateRequest payload ([..., ...$overrides]). The request is "correct from birth" — one INSERT, no post-hoc edit — which is exactly what the spec's "one create, correct from birth" line asked for. Replies (the matched-existing-request branch) never call it, so they bypass mail rules as HelpSpot does. The no-rules path returns [] and the spread is a no-op, so the entire Inbox suite (27 tests) stayed green.
Alternative
Keep the InboundEmail → InboundEmail shape and add mutable override fields to the DTO. Rejected: InboundEmail is a final readonly parse-once DTO (the single Resend parse point), and bolting request-shaped mutable state onto it would blur its single responsibility and break its immutability contract.

Trigger listener uses one class with three handle* methods

Context
The spec lists a single EvaluateTriggers queued listener on three events (RequestCreated, RequestStatusChanged, NoteAdded). Laravel's event auto-discovery (the mechanism this repo relies on — SendPublicReplyEmail has no manual registration) binds a listener by the type-hint of a method named handle — which only maps one event per class.
Decision
Verified in the framework's DiscoverEvents that the match is Str::is('handle*', $method->name) — any public method whose name starts with handle and whose first parameter is an event type is auto-discovered. So EvaluateTriggers has handleRequestCreated / handleRequestStatusChanged / handleNoteAdded, each bound automatically. php artisan event:list confirms all three bindings. No EventServiceProvider, no subscriber registration.
Alternative
An event subscriber (subscribe() method) or three separate listener classes. The subscriber would need manual registration (breaking the zero-config auto-discovery convention); three classes would triplicate the loop-guard + engine-dispatch boilerplate. The handle* convention is idiomatic and keeps the three handlers — which share run() — in one file.

Loop guard works because the sync queue runs nested events in-stack

Context
The guard is a static RuleEngine::$applying flag set by ActionApplier. EvaluateTriggers is ShouldQueue. If the listener ran in a separate worker process (real async queue), the static flag set in the web process wouldn't be visible to the worker, so the guard would appear not to work.
Decision
Kept the listener queued (per spec) and the guard static. The guard is effective because, under the test/request sync queue, a trigger action that fires its own event dispatches the nested EvaluateTriggers synchronously, inside the apply() call stack — where $applying is still true (it's reset in a finally after all effects). So the nested fire is suppressed. The TriggerTest "loop guard" case proves a self-triggering status rule transitions exactly once. Documented as single-level, sync-dependent suppression in the tour doc's "Known limitations" — it stops a rule re-firing itself, not arbitrary cross-rule chains on a fully async queue.
Alternative
Make EvaluateTriggers non-queued (synchronous) to guarantee in-process execution. Rejected: the spec explicitly calls for a queued listener, and the sync-queue test context already exercises the guard correctly. A persistent loop-tracking column or cache key would be the production-grade fix; it's named as future work rather than built, matching the repo's "teach the simple version, name the gap" stance.

Activity-log causation via a marker entry, not a causer

Context
The spec wants the request history to show "what automated the change" via activitylog causedBy/description. But automation has no User causer, and the Phase 2 action classes the applier calls already log their own field diffs through Request's LogsActivity trait.
Decision
ActionApplier::apply() writes one extra activity entry before running the effects: activity()->performedOn($request)->withProperties(['cause' => $cause])->log($cause), where $cause is the rule's layer label + name (e.g. "Mail Rule: Billing keyword routing"). This marker sits alongside the automatic field diffs, so the history reads both what changed (the diffs) and what automated it (the marker) — and it appears even for notify-only rules that produce no field diff. No causer model needed.
Alternative
A dedicated "system" User to act as the causer. Rejected: the repo deliberately keeps no system user (the inbound pipeline made the same call for its "attachment skipped" notes), and a marker entry is simpler and more legible than a synthetic user row.
