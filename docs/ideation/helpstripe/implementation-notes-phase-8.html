CategoryPerformance / AgentPerformance return value objects, not associative-array rows

Context
The spec says each query class returns "row collections matching their table columns." The natural first cut was Collection<int, array{name: string, count: int, …}> — a collection of array shapes. It worked at runtime and the tests passed, but composer test's phpstan step rejected it with a return.type error whose expected and actual types were textually identical, tagged "Template type TValue on Collection is not covariant." phpstan treats array shapes in a collection's value position as invariant, so the declared and inferred types never unify.
Decision
Introduced two readonly value objects — App\Data\CategoryReport and App\Data\AgentReport — exactly mirroring the existing App\Data\UserTeam convention, and returned Collection<int, CategoryReport> / Collection<int, AgentReport>. A named class in the value position is covariant where an array shape is not, so phpstan is satisfied without an @var override, a cast, or a baseline entry. The Blade view reads $row->name instead of $row['name'], and the tests assert against properties. RequestVolume and QueueSnapshot keep their plain-array returns — they're consumed positionally (the chart) and by fixed keys (the cards), not collected, so the invariance issue never arises there.
Alternative
Keep the array shapes and silence phpstan. Rejected: the project's phpstan instructions explicitly forbid @var overrides, casts-to-silence, and baseline suppression — and the codebase already had the value-object pattern (UserTeam) for precisely this "small structured record returned in a typed collection" case, so following it is the idiomatic fix, not a workaround.

The SLA scopes bind "now" as a parameter instead of using the database clock

Context
scopeSlaOverdue needs "minutes since the request was created, measured to now." The obvious SQL is strftime('%s', 'now') (SQLite) / UNIX_TIMESTAMP() (MySQL). But that reads the database's wall clock — and the overdue test freezes time with CarbonImmutable::setTestNow('2026-06-10 12:00:00') while creating a request "30 minutes ago." The DB clock returns the real 2026 instant, so a not-yet-overdue request looked wildly overdue and the test failed.
Decision
"Now" is passed as a bound ? parameter — now()->getTimestamp(), which respects setTestNow — rather than the database's own clock. Only the per-row created_at epoch conversion stays in SQL (the part that genuinely needs the driver branch). The breach math still runs in the database (a real ->count(), no hydration), but it measures elapsed time against the same instant the rest of the app sees.
Alternative
Use the DB clock and special-case tests. Rejected: a query that behaves differently under a frozen clock than the rest of the app is a latent bug, not just a test inconvenience — bound "now" is simply correct.

DemoSeeder needed no change — the existing spread is already non-degenerate

Context
The spec's modified-files table lists DemoSeeder: "verify spread produces non-degenerate charts (some breaches, some in-SLA, all agents represented) — adjust factory weights if needed."
Decision
Verified against the seeded data via the four query classes and made no seeder change. The Phase-1 firstResponseMinutes() helper already alternates inside-SLA (target/2) and breach (target*3) per request, so Billing and Technical Support each show both hits and breaches; Sales (no SLA) shows 0 breaches with a null target; all three frontline staff carry activity; and the Administrator (Sam) holds no assigned requests, which renders as the idle-agent zero row — a happy accident that demonstrates exactly the edge case the spec flagged. Snapshot against the 90-day window: open 20, unassigned 5, urgent 2, breached 14, overdue 7.
Alternative
Force Sam to own some requests so every agent row is non-zero. Rejected: the zero row is the more instructive demo (it proves idle agents appear), and changing the seeder would have disturbed DemoSeederTest's exact-count assertions for no gain.
