API intake — which team a request lands on

Context
The spec describes POST /api/v1/requests with a single static bearer token but doesn't say which tenant/team a request belongs to. CreateRequest derives the team from the customer ($customer->team_id), so the customer must be resolved against some team first.
Decision
Resolve the team as Team::orderBy('id')->first() — the installation's team — exactly as the inbound email pipeline falls back to the first mailbox. The repo's standing decision is "one seeded team = the installation," and the static token has no tenant context of its own. The category_id validation rule is scoped to this team so cross-tenant ids 422.
Alternative
Carry a team id (or per-team token) in config. Rejected: it adds multi-tenant machinery the teaching repo doesn't otherwise have, and the tour doc already names Sanctum as the production path where per-client scoping belongs.

mail:replay — faking the attachment binary fetch offline

Context
ProcessInboundEmail downloads attachment binaries from Resend's short-lived CDN download_url (distinct from the api.resend.com content/listing endpoints). The spec's offline-replay requirement means the command must not hit the network at all, but the spec only mentions faking "the Resend content/attachment endpoints."
Decision
The command reads each download_url straight out of the fixture's attachment listing and registers an Http::fake stub for it returning placeholder PDF bytes — alongside the content and listing stubs. Per-fixture exact-URL stubs (rather than a * wildcard) avoid the wildcard intercepting the next fixture's content fetch when replaying all fixtures in one run.
Alternative
A catch-all '*' stub for attachment bytes. Rejected: Http::fake matches in declaration order and a trailing * from one fixture's fake() call leaked into the next replay, returning a non-JSON body where the email content was expected and tripping the job's array return type.
