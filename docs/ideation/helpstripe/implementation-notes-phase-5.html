Implementation Notes — Phase 5

HelpStripe · 2026-06-10 · Knowledge Base

Portal layout — created by Phase 5, not Phase 4

Context
The spec's pattern reference for portal pages is "Phase 4 portal pages (layout + public routes)", but Phase 5 ran first (it depends only on Phase 1). The spec sanctions this: "whichever phase lands second completes the wiring".
Decision
Created the minimal resources/views/layouts/portal.blade.php here: brand header, footer, no app chrome, with Route::has('portal.home') / Route::has('portal.kb.index') guards in the header so navigation lights up as phases land. Phase 4 must extend this file, not recreate it (its File Changes table lists it as a new file).
Alternative
Rendering portal KB pages in the app layout — rejected: the layout contrast is a named lesson, and the app layout assumes an authenticated user.

Routes — portal group must precede the {current_team} group

Context
The existing {current_team} prefix group matches any first URL segment, so /portal/kb was captured as team "portal" and redirected guests to login (caught by PortalBrowsingTest, 302s).
Decision
Registered the portal route group before the {current_team} group; first match wins. A comment in routes/web.php records both ordering constraints (portal-before-team and search-before-book-slug). Phase 4's portal routes must join this same group position.
Alternative
A where() regex constraint excluding "portal" from {current_team} — rejected: ordering is simpler and Phase 4 adds more portal routes anyway.

Admin routes — flat id-bound editor URL, slugs only on the portal

Context
The spec specifies portal URLs precisely (nested slugs + scopeBindings()) but is silent on admin URLs for the book manager and page editor.
Decision
Admin binds by id: kb/{book} and a flat kb/pages/{page} for the editor. The scoped-binding lesson lives on the portal where it matters; admin mount() does an explicit cross-team check instead.
Alternative
Mirroring the nested slug URLs in admin — rejected: renames would relocate the editor URL mid-edit, and the duplication teaches nothing new.

Cross-team manager access — 404 via mount() check, not a policy

Context
spatie roles are global (teams feature off), so an Administrator of team A technically holds 'manage knowledge base' everywhere; the spec doesn't address cross-team ids reaching the manager.
Decision
Every admin mount()/action verifies the record belongs to current_team_id and 404s otherwise (existence not leaked). Contrast documented in the tour doc against Phase 2's policy-based 403.
Alternative
A KnowledgeBookPolicy — rejected: the phase's stated lesson is permission middleware vs. policies; adding a policy would blur the contrast.

Search — explicit ESCAPE clause for LIKE wildcards

Context
The spec's failure mode table says to escape %/_ in the search term. Backslash-escaping alone is a MySQL convention — SQLite (the test database) has no default LIKE escape character, so \% would not work there.
Decision
whereRaw('title LIKE ? ESCAPE ?', [$like, '\\']) with addcslashes($term, '\%_') — portable across SQLite, MySQL, and Postgres. Covered by the wildcard-literal test in SearchTest.
Alternative
Laravel's whereLike() — rejected: it does not escape wildcards in the value, which is the entire point here.

Portal home KB teaser — deferred to Phase 4

Context
The spec's Modified Files lists pages/portal/⚡home.blade.php "Activate KB teaser (if Phase 4 already shipped)". Phase 4 has not shipped; the file does not exist.
Decision
Nothing to wire — Phase 4 carries the same Route::has('portal.kb.index') guard and will complete the teaser when it creates the home page. The portal layout's header already links to the KB for navigation in the meantime.
Alternative
Creating a stub portal home — rejected: out of scope; that page is Phase 4's.
