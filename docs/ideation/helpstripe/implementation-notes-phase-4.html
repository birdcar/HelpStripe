Implementation Notes — Phase 4

HelpStripe · 2026-06-10 · Self-Service Portal

Confirmation mail sent from the submit component, not CreateRequest

Context
The spec's "Modified Files" lists app/Actions/Requests/CreateRequest.php with "Ensure source=Portal path sends NewRequestConfirmationMail (queued)". But the existing email pipeline keeps the mail-send out of the action: ProcessInboundEmail (the job) calls CreateRequest and then sends the mail itself. The action is the shared write-path used by the agent UI, API, email, and now the portal.
Decision
Left CreateRequest untouched and queued NewRequestConfirmationMail from ⚡submit, mirroring ProcessInboundEmail line-for-line. Putting the send in the action would have fired a confirmation email on every agent-created and API-created request too — the agent UI files requests on customers' behalf and must not blast them a "we received your request" email. The channel that owns the customer relationship sends the mail; the action just opens the request.
Alternative
A source === Portal branch inside CreateRequest — rejected: channel-specific side effects in a channel-agnostic action is exactly the coupling the shared write-path exists to avoid, and it contradicts the established Phase 3 split.

Throttle sits on the page-load route, not the Livewire form POST

Context
The spec wants throttle:10,1 on submit/lookup and the experiment reads "11th submit in a minute → 429". But the forms are Livewire components: the submit action is an AJAX POST to /livewire/update, not to the portal/submit route. Livewire only re-applies a fixed set of persistent middleware to update requests, and throttle isn't one (persistent middleware can't take arguments, per the Livewire 4 security docs).
Decision
Kept throttle:10,1 on the GET routes (it throttles page loads — the honest thing the middleware actually does) and wrote SubmitRequestTest to assert the 11th GET /portal/submit in a minute returns 429. The tour doc (§2) explains the page-load-vs-POST nuance explicitly rather than papering over it.
Alternative
A custom rate limiter applied via Livewire::setUpdateRoute() or a per-component throttle on the action — rejected: global to all Livewire updates (wrong blast radius) or substantially more machinery than a teaching repo's "throttle, taught explicitly" warrants. Route-level throttling is the idiomatic, documented approach.

One status component, two routes (signed + session-verified)

Context
The spec describes "two access paths to the same status page" — a signed URL from the email and a manual lookup — but both must land on "one route handler". A single route can't carry the signed middleware and be reachable without a signature (the manual-lookup landing).
Decision
Registered two routes pointing at the same ⚡status component: portal.status (with signed middleware — the email link) and portal.status.show (unsigned — the manual-lookup landing). The component's mount() grants a session flag portal.verified.{id} on a valid signature, then abort_unlesses on that flag — so the unsigned route is gated by the session a prior signed visit or manual lookup set. Same handler, two doors.
Alternative
One route with conditional middleware, or stuffing the access key into the URL — rejected: conditional middleware can't be expressed cleanly on a Livewire route, and an access key in the URL would land in browser history and server logs. The signature is the credential for the link path; the session flag for the lookup path.

Portal pages read the installation team via the same fallback as the API

Context
The portal is unauthenticated — there's no current_team_id to scope categories or customer lookups by. The spec assumes "one seeded team = the installation" but doesn't say how the public pages reach it.
Decision
Used the identical fallback the inbound email pipeline and API already established: Team::query()->orderBy('id')->first() (see StoreRequestRequest::installationTeam()). The submit page's category select and the lookup's request resolution both scope to that team. Inlined the one-liner in each portal component rather than extracting a shared helper — it's three call sites and the existing API code already duplicates it; a premature abstraction would be the only new thing to learn.
Alternative
A config('helpstripe.team_id') or a global scope — rejected: the order-by-id-first fallback is already the codebase's documented convention for "the installation", and matching it keeps the three public channels consistent.
